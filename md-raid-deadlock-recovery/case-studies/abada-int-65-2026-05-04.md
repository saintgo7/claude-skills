## 6. 데이터 손실 가능성 평가

- in_flight = 0 → 디스크에 보내진 IO는 없으므로 **이미 디스크에 쓰인 데이터는 안전**
- 27분간 hang한 COMMIT 2건 (블로그 신규 사용자 INSERT) → 미커밋 상태로 롤백됨, 사용자 재가입 필요
- 큐에 쌓인 INSERT 7건 → 마찬가지로 롤백
- 페이지 캐시에서 dirty인데 못 flush한 데이터: 재부팅 시 손실 가능. 단 대부분의 컨테이너는 fsync로 critical data를 즉시 내리므로 영향 작음

## 8. 근본 원인 (kernel bug 의심)

`raid5_get_active_stripe` ↔ `kthread_stop` 데드락은 Linux md/raid5 드라이버에서 알려진 패턴. 일반적 트리거:
1. mdcheck 진행 중 (정합성 검사) sync_action을 동시에 변경
2. 매우 큰 array(28TB)에서 stripe cache 부족
3. 백그라운드 IO 부하와 동시 발생

**예방:** mdcheck timer 빈도 줄이거나 비활성화 + stripe cache 크기 조정 (`/sys/block/md0/md/stripe_cache_size`, 기본 256, 권장 4096~16384).

### 9-A. 컨테이너 자동기동 — 결론: ✅ 안전

```
restart 정책별 컨테이너 수:
  always         97개   ← 자동기동 ✓
  unless-stopped 39개   ← 자동기동 ✓ (직전에 stop된 것만 안 올라옴)
  no              6개   ← 자동기동 ✗
```

`no`인 6개는 익명 테스트 컨테이너(`infallible_vaughan`, `exciting_driscoll`, `vigorous_golick`, `quizzical_mccarthy`, `dreamy_hypatia`, `angry_galileo`) — 운영 서비스 아님, 무시 가능.

⚠️ **주의:** 재부팅 직전 절차 5-2의 `docker stop`을 모든 컨테이너에 일괄 실행하면 `unless-stopped` 정책의 39개도 stopped 상태가 됨 → 부팅 후 자동기동 안 됨. 그러나 `docker start <name>`으로 일괄 재기동 가능. 또는 stop 단계에서 **always 정책만 빼고** 정지하는 방식도 가능.

권장: 그냥 호스트 `reboot`만 하면 dockerd가 SIGTERM으로 컨테이너 정상 종료 → 모두 자동 기동. 절차 5-2의 명시적 `docker stop`은 생략 가능 (단, fsync 대기로 reboot가 길어질 위험은 있음).

### 9-B. RAID 자동 재검사 차단 — 사전 작업 필요 ⚠️

현재 시스템에 다음이 활성화돼 있어 그대로 두면 부팅 직후 동일 데드락 재발 가능:

| 항목 | 상태 | 영향 |
|---|---|---|
| `/etc/default/mdadm` `AUTOCHECK=true` | true | 부팅 시 자동 정합성 검사 |
| `mdcheck_start.timer` | enabled (다음 6/7) | 매월 검사 시작 |
| `mdcheck_continue.timer` | disabled지만 자동 재활성 | 검사 미완료 시 hourly |
| `/var/lib/mdcheck/MD_UUID_c8eb...` | 존재 | continue가 이 파일 보고 트리거 |
| `stripe_cache_size` | 256 (기본) | RAID 캐시 부족이 데드락 트리거 중 하나 |

**재부팅 전 실행 (모두 sda6 = / 위 작업이라 데드락과 무관, 안전):**

```bash
# 1. mdcheck systemd timer 둘 다 정지/비활성
sudo systemctl stop mdcheck_start.timer mdcheck_continue.timer
sudo systemctl disable mdcheck_start.timer mdcheck_continue.timer

# 2. continue trigger 파일 백업 후 제거
sudo cp /var/lib/mdcheck/MD_UUID_c8ebb2e9:2e2b4314:148f6489:c88cd7df \
        /root/MD_UUID.bak.$(date +%s)
sudo rm /var/lib/mdcheck/MD_UUID_*
sudo rm /var/lib/mdcheck/.md-check-*

# 3. /etc/default/mdadm 백업 후 AUTOCHECK 차단
sudo cp /etc/default/mdadm /etc/default/mdadm.bak.$(date +%s)
sudo sed -i 's/^AUTOCHECK=true/AUTOCHECK=false/' /etc/default/mdadm

# 4. (확인) sync_action=frozen 그대로 두기 — 변경 시 셸 hang
cat /sys/block/md0/md/sync_action   # frozen 이어야 함
```

원복 시점: 데이터 무결성 검증 + stripe_cache_size 튜닝 후 재활성화.

echo idle | sudo tee /sys/block/md0/md/sync_action

# 3. stripe_cache_size 상향 (재발 방지)
echo 8192 | sudo tee /sys/block/md0/md/stripe_cache_size
# 영구 적용은 /etc/sysctl 아닌 별도 systemd unit 필요 — 일단 임시로

# 4. /data 마운트 + 쓰기 테스트
mountpoint /data && touch /data/.write_test_$(date +%s) && rm /data/.write_test_*

# 5. 컨테이너 상태
docker ps -a --format 'table {{.Names}}\t{{.Status}}' | grep -v "Up " | head
docker ps | wc -l    # 130개 정도 기대

# 6. 핵심 서비스 헬스체크
for url in https://blog.abada.co.kr https://argos.abada.co.kr \
           https://shop.abada.kr https://safe.abada.kr; do
  echo -n "$url -> "
  curl -sI -o /dev/null -w "%{http_code}\n" -m 10 "$url"

## 11. 비상 시나리오

| 증상 | 대응 |
|---|---|
| `reboot` 명령이 hang | 다른 셸에서 `echo b > /proc/sysrq-trigger` (즉시 강제재부팅) |
| 부팅 후 SSH 미응답 (10분+) | `cloudflared access ssh --hostname s65.abada.co.kr` 시도 |
| 위까지 다 실패 | 서버실 물리 콘솔 (지난번 경로) |
| 부팅 후 RAID 또 frozen | mdadm 메시지 `dmesg | tail -50` 확인, 9-B의 모든 항목이 적용됐는지 재확인 |
| 부팅 후 컨테이너 다수 unhealthy | 5분 대기 (dependency 부팅 순서). 그래도 안 되면 `docker compose -f /data/{path}/docker-compose.yml up -d` 개별 |
